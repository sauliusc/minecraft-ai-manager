# One-Click Docker Deployment — Architecture

## Goal

Deploy the full Minecraft AI Manager stack with a **single command**:

```bash
docker compose up -d
```

No wizard. No `make setup`. No separate migration step. No manual admin seeding.

---

## What changes vs deploymentV2

| Topic | deploymentV2 (current) | deployDocker (new) |
|---|---|---|
| First-time setup | `make setup` (interactive wizard) then `make deploy` | `docker compose up -d` |
| Secrets | Generated by wizard, written to `.env` | Pre-generated in committed `.env`; clearly marked "change for production" |
| DB migrations | Run externally by `deploy.sh` via `docker compose exec` | Run inside the API container on every startup |
| Admin account | `make seed-admin` (prompts for email/password) | Created automatically from `ADMIN_EMAIL` / `ADMIN_PASSWORD` env vars on first start |
| Target user | Junior devops with some CLI experience | Complete beginners; anyone with Docker installed |

## Key design decisions

### 1. Secrets in `.env` committed to git

The `.env` file ships with **pre-generated secrets** so the stack works immediately. Every secret has a prominent comment warning the user to change it before exposing the server to the internet.

This is the standard approach for developer-friendly Docker projects (e.g. Supabase, Pocketbase). For a home/LAN Minecraft server it is safe enough by default.

Users who want stronger security can regenerate secrets with one command documented in the README.

### 2. Migrations inside the API entrypoint

The `server/Dockerfile` gains an `entrypoint.sh` that runs before `node dist/index.js`:

```
entrypoint.sh
├── npx prisma migrate deploy   ← applies pending migrations
│   └── (fallback) npx prisma db push   ← if no migration files
├── node scripts/seed-admin.js  ← creates admin if ADMIN_EMAIL is set and no users exist
└── exec node dist/index.js     ← starts the API
```

This means migrations are always in sync with the running image, and there is no external step to forget.

### 3. Auto admin seeding via env vars

Two env vars control first-time admin creation:

```
ADMIN_EMAIL=admin@example.com
ADMIN_PASSWORD=changeme123
```

The seed script is idempotent: if a user with that email already exists, it does nothing. The plain-text password in `.env` is a one-time bootstrap credential — users should change it after first login.

### 4. One file to rule them all

A new `docker-compose.yml` lives at the **repository root** (not inside `deploymentV2/`). This is the file Docker Compose finds by default — no `-f` flag needed.

The existing `deploymentV2/` is kept for power users who want rollback, backup, and Makefile commands.

## `.env` structure

```dotenv
# ════════════════════════════════════════════════════════════════
#  Minecraft AI Manager — default configuration
#  ⚠  CHANGE ALL SECRETS BELOW BEFORE EXPOSING TO THE INTERNET ⚠
# ════════════════════════════════════════════════════════════════

# Panel port
HTTP_PORT=80

# Database
POSTGRES_DB=craftcontrol
POSTGRES_USER=craftcontrol
POSTGRES_PASSWORD=<pre-generated>

# Redis
REDIS_PASSWORD=<pre-generated>

# JWT
JWT_SECRET=<pre-generated>
JWT_REFRESH_SECRET=<pre-generated>
JWT_EXPIRES_IN=15m
JWT_REFRESH_EXPIRES_IN=7d

# Minecraft bridge
BRIDGE_SECRET=<pre-generated>       # copy this to your Minecraft server
GAME_VM_HOST=127.0.0.1              # IP of your Minecraft server

# First-run admin account (change password after first login)
ADMIN_EMAIL=admin@example.com
ADMIN_PASSWORD=changeme123
```

## docker-compose.yml structure

Same 4 services as `deploymentV2`. Key additions:

- `api` service uses `entrypoint.sh` instead of bare `CMD`
- `ADMIN_EMAIL` / `ADMIN_PASSWORD` passed to API
- `restart: unless-stopped` on all services
- No external `.env` path flag — uses root `.env` automatically

## Startup sequence

```
docker compose up -d
        │
        ├─▶ db (postgres)  ──healthcheck──▶ ready
        ├─▶ redis          ──healthcheck──▶ ready
        │                                    │
        └─▶ api  (depends_on db+redis healthy)
                │
                ├─ entrypoint.sh
                │   ├─ prisma migrate deploy
                │   ├─ seed-admin.js (once)
                │   └─ node dist/index.js ──healthcheck──▶ ready
                │                                            │
                └─▶ web  (depends_on api healthy)
                    └─ nginx serves SPA + proxies /api/
```

Total time from `docker compose up -d` to panel accessible: **~60 seconds** on a cold start.

## Security notes for production

1. Change `POSTGRES_PASSWORD`, `REDIS_PASSWORD`, `JWT_SECRET`, `JWT_REFRESH_SECRET`, `BRIDGE_SECRET` in `.env`
2. Change `ADMIN_PASSWORD` and remove `ADMIN_EMAIL` / `ADMIN_PASSWORD` from `.env` after first login
3. Place behind a reverse proxy (Nginx/Caddy) with TLS — see `deploymentV2/README.md` for Cloudflare / Let's Encrypt options
4. Restrict port 80 to the reverse proxy; do not expose PostgreSQL or Redis ports
